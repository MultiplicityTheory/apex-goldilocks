# ADR 001: Unification of Atlas, Multiplicity-Core, and Monstrous Moonshine (Apex)

## Status
Proposed

## Context
The Apex project currently consists of several fragmented components:
1. **Atlas-Embeddings**: A high-integrity Rust engine for exceptional Lie group construction using exact arithmetic.
2. **Multiplicity-Core**: A Rust-native AEP (Atlas-Embedding Proof) system, but with some legacy dependencies on floating-point arithmetic and Python plugins.
3. **Monstrous Moonshine**: Research-level code (Python and Rust) for modular symmetry and ZK-sketches, currently split between floating-point simulations and integer-scaled proofs.

The "Sedona Spine Mandate" (GEMINI.md) requires a single mandatory source of truth in Rust with "Zero Drift" and "No Floats".

## Decision
We will unify these components into a single Rust workspace named **Apex**. 

### 1. Workspace Structure
Apex will be a Cargo workspace with the following members:
- `apex-core`: Foundational arithmetic (Rational/HalfInteger), Atlas graph, and E8 root systems (Ported from `atlas-embeddings`).
- `apex-aep`: Core AEP logic including ACE Runtime, PETC Ledger, PIRT-M (Power Iteration Resonance Tensor Monitor), and SPASC (Scaled Projected Alternating Sub-gradient Confinement).
- `apex-moonshine`: Monstrous Moonshine embeddings, modular operator definitions, and ZK-sketching for feedback interconnection.
- `apex-cli`: A unified CLI for construction, verification, and audit bundle generation.

### 2. Arithmetic Mandate
- **All** floating-point (`f64`) logic in AEP and Moonshine will be replaced with exact scaled integer (`i64`/`i128`) or `Rational` arithmetic.
- Transcendental functions (like Bessel functions in Moonshine) must be replaced with rational approximations or deterministic integer-scaled lookups to maintain proof reproducibility.

### 3. Proof Hashing
- Standardize on **BLAKE2b-256** for all proof digests and `proof_id` generation.

### 4. Integration Path
- The `Engine (Rust)` will be the sole source of truth.
- Python and other languages will interface via a WASM-compiled SDK or the unified CLI.

## Consequences
- **Integrity**: Every proof generated by Apex will be mathematically exact and formally verifiable.
- **Portability**: The Rust core will be `no-std` compatible where possible, allowing execution in constrained environments (WASM/TEE).
- **Maintenance**: A single workspace simplifies dependency management and ensures cross-component compatibility (e.g., Moonshine operators using Atlas primitives).
